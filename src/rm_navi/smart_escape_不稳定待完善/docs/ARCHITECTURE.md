## 关键设计决策

### 1. 为什么使用独立 Action Server？

**问题：插件模式的问题**
```
bt_navigator (Activating)
    │
    ├── 扫描XML中的所有节点
    ├── 发现 <BackUp> 节点
    ├── 尝试与 recoveries_server 握手
    │       └── 插件正在初始化...
    │           └── 等待地图回调...
    │               └── 阻塞！
    └── 超时/卡住
```

**解决方案：独立 Action Server**
```
bt_navigator (Activating)
    │
    ├── 扫描XML中的所有节点
    ├── 发现 <SmartEscape> 节点
    ├── BT节点只是轻量级封装
    │   └── 不需要等待任何服务
    └── 启动成功！

smart_escape_server (独立进程)
    │
    ├── 独立初始化
    ├── 订阅代价地图
    └── 等待Action请求
```

### 2. 为什么 BT 节点不直接访问代价地图？

| 方案 | 优点 | 缺点 |
|------|------|------|
| BT节点直接访问代价地图 | 简单直接 | BT节点变重，可能阻塞行为树执行 |
| **独立Server访问代价地图** | BT节点轻量，Server可以复杂计算 | 需要Action通信开销 |

选择独立Server的原因：
- 脱困算法可能需要较复杂的计算
- 避免阻塞行为树的实时响应
- 符合Nav2的设计哲学（服务分离）

### 3. 川大算法的核心创新

**传统方法的问题**：
```
机器人被卡住
    │
    ├── 旋转（随机角度）
    ├── 后退（固定距离）
    └── 重新规划路径
        └── 可能还是会被卡住！（没有方向性）
```

**川大智能脱困**：
```
机器人被卡住
    │
    ├── 分析代价地图
    │   └── 找到"自由空间"的方向
    ├── 计算质心
    │   └── 确定最优脱困方向
    └── 朝向自由空间移动
        └── 有方向性的脱困！
```

## 节点通信流程

### 正常导航流程

```
[User] ──► [bt_navigator] ──► [ComputePathToPose] ──► [planner_server]
                                          │
                                          ▼
[User] ◄── [FollowPath] ◄── [controller_server] ◄── [bt_navigator]
```

### 脱困触发流程

```
[FollowPath] 失败
    │
    ▼
[RecoveryNode] 触发恢复
    │
    ▼
[SmartEscape BT Node] ──Action──► [smart_escape_server]
    │                                    │
    │                                    ├── 获取代价地图
    │                                    ├── 计算脱困目标
    │                                    └── 返回结果
    │                                    │
    ▼                                    ▼
[ComputePathToPose] ◄── [escape_pose] ──┘
    │
    ▼
[FollowPath] 移动到脱困目标
```

## 性能考虑

### 计算复杂度

```
搜索半径 r，栅格分辨率 res

栅格数 ≈ π × (r/res)²

例如：r=0.5m, res=0.05m
栅格数 ≈ 3.14 × (10)² ≈ 314 个栅格

这是非常轻量的计算！
```

### 实时性保证

```
┌────────────────────────────────────────────────────────────┐
│  行为树执行周期: 20Hz (50ms)                                 │
│                                                             │
│  SmartEscape BT Node:                                      │
│  ├── 发送Action请求: <1ms                                   │
│  ├── 等待结果: 10-50ms (异步，不阻塞其他节点)                 │
│  └── 返回RUNNING: <1ms                                      │
│                                                             │
│  smart_escape_server:                                      │
│  ├── 接收请求: <1ms                                         │
│  ├── 计算脱困目标: 5-20ms                                   │
│  └── 返回结果: <1ms                                         │
│                                                             │
│  总延迟: 15-70ms << 行为树周期，满足实时性要求               │
└────────────────────────────────────────────────────────────┘
```

## 扩展建议

### 1. 多方向脱困

当前算法只选择一个最优方向，可以扩展为：
```
计算多个候选方向
    ├── 方向1: 质心角度 + 0°
    ├── 方向2: 质心角度 + 45°
    └── 方向3: 质心角度 - 45°
选择代价最小的方向
```

### 2. 历史信息融合

```
记录最近N次脱困位置
    └── 避免重复访问同一区域
```

### 3. 动态参数调整

```
根据卡住频率动态调整:
    ├── 频繁卡住 → 增大 escape_distance
    └── 脱困失败 → 增大 max_radius
```
